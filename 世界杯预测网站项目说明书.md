# 世界杯比赛预测移动端网站项目说明书

## 1. 项目背景

本项目面向移动端用户，围绕“世界杯比赛预测”内容搭建一个专题型网站。网站主要展示每场比赛预测、每轮赛事专题，以及球队历史预测记录，方便用户通过手机快速浏览、分享和回看。

## 2. 项目目标

### 2.1 核心目标

- 建设一个适合手机访问的世界杯预测内容网站。
- 将 Markdown 形式的预测素材转化为结构化网页。
- 支持按比赛、轮次、球队查看预测内容。
- 支持后续持续更新素材并快速重新发布。
- 发布到互联网，生成可分享链接，方便朋友访问。

### 2.2 体验目标

- 页面视觉要有体育赛事氛围，专业、紧凑、有冲击力。
- 移动端首屏必须清晰展示重点比赛和预测结论。
- 内容阅读体验要接近体育资讯 App，而不是普通文档站。
- 操作路径要短，用户进入网站后能快速找到感兴趣的比赛或球队。

## 3. 目标用户

### 3.1 主要用户

- 通过微信、朋友圈、群聊打开链接的朋友。
- 对世界杯比赛预测感兴趣的球迷。
- 希望快速查看某场比赛倾向、比分预测和分析理由的轻度用户。

### 3.2 用户特点

- 主要使用手机访问。
- 不愿意阅读太长的纯文本。
- 更关注结论、信心指数、比分预测和关键理由。
- 容易通过社交平台转发单场专题页。

## 4. 产品范围

### 4.1 第一版必须包含

1. 首页
   - 展示热门比赛预测。
   - 展示最新更新内容。
   - 展示当前轮次入口。
   - 展示热门球队入口。

2. 单场比赛专题页
   - 比赛双方。
   - 比赛时间。
   - 所属轮次。
   - 胜平负预测。
   - 比分预测。
   - 信心指数。
   - 关键分析内容。
   - 同轮其他比赛推荐。
   - 相关球队历史预测入口。

3. 每轮专题页
   - 展示该轮全部比赛预测。
   - 标记重点比赛。
   - 支持从轮次页面进入单场专题。

4. 球队历史预测页
   - 展示某支球队相关的全部历史预测。
   - 按时间或轮次排序。
   - 支持进入每场比赛详情。

5. 内容导入机制
   - 从指定 Markdown 素材目录读取内容。
   - 不修改原素材目录。
   - 自动生成网站所需结构化数据。

6. 移动端适配
   - 以 375px 到 430px 手机宽度为主要设计范围。
   - 同时兼容平板和桌面浏览。

### 4.2 第二版可扩展

- 搜索功能。
- 按球队、轮次、日期筛选。
- 命中率统计。
- 比赛结束后补充实际赛果。
- 首页增加“高信心推荐”模块。
- 专题页增加分享封面图。
- 增加后台或轻量 CMS。

## 6. 页面设计要求

### 6.1 首页

首页是最重要的入口，需要在首屏呈现“今天最值得看的预测”。

建议模块：

- 顶部赛事标题区。
- 焦点比赛大卡片。
- 横向轮次切换栏。
- 最新预测列表。
- 热门球队快捷入口。
- 底部导航栏。

首页首屏要展示：

- 网站主题。
- 一场重点比赛。
- 预测结论。
- 信心指数。
- 进入详情的按钮。

### 6.2 单场比赛专题页

单场页是分享传播的核心页面。

建议结构：

- 顶部对阵区：球队 A vs 球队 B。
- 比赛基础信息：时间、轮次、场地。
- 预测结果区：胜平负、比分、进球数。
- 信心指数区：用进度条或仪表视觉表达。
- 分析正文区：展示 Markdown 正文。
- 风险提示区：爆冷因素、谨慎点。
- 相关内容区：同轮其他比赛、双方历史预测。

### 6.3 轮次专题页

轮次页用于承载一个阶段的整体内容。

建议结构：

- 轮次标题。
- 轮次概览。
- 重点比赛推荐。
- 本轮比赛列表。
- 晋级/排名/形势分析入口，视素材情况决定。

### 6.4 球队历史预测页

球队页用于沉淀内容，让用户可以回看某队相关预测。

建议结构：

- 球队名称。
- 预测记录数量。
- 最近预测。
- 历史预测时间线。
- 对阵对手与预测结论。

## 7. 内容结构设计

Markdown 素材需要被转为结构化数据。理想情况下，每篇 Markdown 对应一篇专题内容。

### 7.1 推荐字段

```yaml
title: 阿根廷 vs 法国 比赛预测
type: match
round: 小组赛第一轮
teams:
  - 阿根廷
  - 法国
matchDate: 2026-06-18
prediction: 阿根廷不败
scorePrediction: 2-1
confidence: 78
tags:
  - 焦点战
  - 高热度
```

### 7.2 内容类型

建议支持三种内容类型：

- `match`：单场比赛预测。
- `round`：轮次专题。
- `team`：球队历史或球队专题。

### 7.3 如果原 Markdown 不规范

如果现有 Markdown 没有统一 frontmatter，可以采用两种方式：

1. 通过文件名和文件夹结构推断比赛、球队、轮次。
2. 在项目中维护一个独立映射文件，例如 `content-map.json`。

注意：映射文件属于网站项目，不放入原素材目录。

## 8. 内容更新机制

### 8.1 基本原则

- 原始 Markdown 文件夹只读。
- 网站项目拥有自己的构建数据。
- 每次内容更新后，通过脚本重新生成页面数据。

### 8.2 推荐流程

```text
1. 将新的 Markdown 素材放入原素材目录
2. 在网站项目中运行内容导入脚本
3. 脚本读取 Markdown 并生成结构化 JSON
4. 本地预览网站
5. 确认无误后构建生产版本
6. 发布到线上
```

### 8.3 推荐命令设计

```bash
npm run import-content
npm run dev
npm run build
```

### 8.4 自动生成数据

导入脚本建议生成：

- `src/data/matches.json`
- `src/data/rounds.json`
- `src/data/teams.json`
- `src/data/articles.json`

这样前端页面可以直接读取这些 JSON 数据，不需要数据库。

## 9. 技术方案建议

### 9.1 推荐技术栈

第一版推荐：

- Vite
- React
- TypeScript
- Markdown 解析器
- 本地导入脚本
- 静态部署平台

选择理由：

- 开发速度快。
- 适合移动端交互。
- 部署简单。
- 后续可以扩展搜索、筛选、统计等功能。

### 9.2 备选方案

如果更重视 SEO 和静态内容生成，可以使用 Astro。

Astro 的优势：

- 静态内容站更自然。
- 页面加载更轻。
- Markdown 内容处理能力强。

React + Vite 的优势：

- 交互体验更好。
- UI 组件开发更灵活。
- 后续做筛选、搜索、动态效果更方便。

本项目第一版建议使用 React + Vite，因为移动端专题站对交互和视觉要求较高。

## 11. 发布方案

### 11.1 推荐平台

第一优先级：Cloudflare Pages

适合原因：

- 静态网站部署方便。
- 免费额度适合个人项目。
- 支持 Git 自动部署。
- 支持自定义域名。
- 全球访问速度较好。

### 11.2 备选平台

Vercel：

- 适合 React/Vite/Next.js 项目。
- 预览部署体验好。
- 使用简单。

GitHub Pages：

- 免费。
- 适合纯静态页面。
- 功能较少，但足够简单项目使用。

### 11.3 推荐上线流程

```text
1. 本地完成开发
2. 创建 GitHub 仓库
3. 上传项目代码
4. Cloudflare Pages 连接 GitHub 仓库
5. 设置构建命令：npm run build
6. 设置输出目录：dist
7. 发布生成 pages.dev 链接
8. 可选：绑定自己的域名
```

## 12. 维护方式

### 12.1 日常更新

内容维护者只需要做三件事：

1. 新增或修改 Markdown 素材。
2. 运行导入和构建命令。
3. 发布到线上。

第一版不需要数据库。
